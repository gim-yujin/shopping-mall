# ADR-0002: 포인트는 배송완료 시 정산하고, 취소/반품은 상태 기반으로 처리한다

- 상태: 채택
- 날짜: 2026-03-14
- 관련 코드: `src/main/java/com/shop/domain/order/service/OrderService.java`, `src/main/java/com/shop/domain/order/service/OrderCancellationService.java`, `src/main/java/com/shop/domain/order/service/PartialCancellationService.java`

## 배경(Context)
- 포인트를 주문 시점에 즉시 적립하면, 미배송 주문의 포인트를 다른 주문에 먼저 사용한 뒤 원 주문이 취소될 때 정합성 문제가 발생할 수 있다.
- 특히 취소 시점의 차감/환불 순서와 경계값 처리(예: 음수 클램핑) 때문에 부당 지급 리스크가 있었다.
- 부분 취소/반품은 전체 취소와 다른 상태 전이/정산 규칙이 필요하다.

## 결정(Decision)
- 적립 포인트는 주문 생성 시 즉시 지급하지 않고, 배송완료(`DELIVERED`) 시점에 정산한다.
- 주문에는 `earnedPointsSnapshot`(적립 예정치)만 저장하고, 실제 지급은 `points_settled=false -> true` 전환 시 1회 수행한다.
- 취소/반품 정책은 아래처럼 상태 기반으로 분리한다.
  - `PENDING`/`PAID` 취소: 적립 포인트 미정산 상태이므로 적립 포인트 차감 없이 사용 포인트 환불만 처리
  - `DELIVERED` 이후 취소/반품: 이미 정산된 적립 포인트를 차감(회수)하고 사용 포인트 환불을 함께 처리
  - 부분 취소/반품: 아이템 비중 기반으로 환불 금액/사용 포인트를 비례 정산

## 대안(Alternatives)
1. 주문 시 즉시 적립(미채택)
   - 장점: 고객 체감이 즉각적
   - 단점: 취소/반품과 결합 시 정합성/멱등성 복잡도 증가, 부당 지급 리스크
2. 배송완료 시 적립 + `points_settled` 플래그(채택)
   - 장점: 상태 전이에 따라 정산 책임이 명확, 중복 정산 방지 용이
   - 단점: 고객이 포인트를 늦게 받는 UX trade-off

## 근거(Why)
- 포인트는 주문의 “완결된 가치 제공(배송완료)” 이후 부여하는 것이 회계/정합성 관점에서 안전하다.
- `points_settled` 기반 멱등 제어는 재시도/중복 호출 상황에서 안정적이다.
- 취소/반품을 상태 기반으로 나누면 케이스별 책임이 명확해지고 테스트 가능성이 높아진다.

## 결과(Consequences)
- 기대 효과:
  - 포인트 부당 지급 및 중복 정산 위험 감소
  - 취소/반품 정책의 일관성 향상
- 감수해야 할 비용/리스크:
  - 배송완료 전 포인트 미사용에 대한 사용자 안내가 필요
- 후속 작업:
  - 관리자/마이페이지 화면에 “적립 예정/정산 완료” 상태를 분명히 노출
  - 취소/반품/부분취소 회귀 테스트를 유지
